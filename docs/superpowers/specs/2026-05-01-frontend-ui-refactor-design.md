# 前端界面重构设计（方案 B：中度重构）

日期：2026-05-01  
范围：`web/frontend`（React + Vite）  
目标页面：转换首页（Convert）、拆零管理（Split Manager）  

## 1. 背景与现状

当前前端已具备较完整功能：模板选择、文件上传、转换进度/结果、日志展示，以及拆零管理的表格编辑与分页。样式已使用 CSS 变量（design tokens 的雏形），交互也较顺畅。

本次重构不以“修问题”为主，而是以 **体验更像工作台、视觉更成品、代码更好扩展** 为主。

## 2. 目标与非目标

### 2.1 目标（Goals）
- **转换流程体验**：信息层级更清楚（操作/状态/日志分离），反馈更稳定（明确 loading/成功/失败/警告），日志可读性更高。
- **拆零管理效率**：批量编辑更顺滑，未保存更改更明确，保存/失败反馈统一。
- **视觉统一与质感**：沉淀更系统的 tokens 与基础组件样式（Button/Input/Modal/Toast/Badge/Card 等），减少页面“各做各的”。
- **不在 main 分支直接改**：后续实现阶段在新分支进行开发。

### 2.2 非目标（Non-goals）
- 不引入重型 UI 框架（如 Ant Design、MUI）作为依赖（保持轻量与可控）。
- 不改变后端 API 协议（仅在需要时做前端适配层）。
- 不在本阶段引入复杂全局状态管理（如 Redux）；仍以组件状态 + 轻量 hooks 为主。

## 3. 总体方案（方案 B：中度重构）

核心策略：**沉淀一层轻量 UI 基础设施（tokens + ui 组件 + 请求封装）**，并对两个核心页面做结构级布局优化（而不是只改 CSS）。

已确认的布局决策：
- 转换首页：**两栏工作台布局（桌面端）**，移动端自动回落单列。
- 拆零管理：**表格工作台布局**（强化批量编辑效率）。

## 4. 信息架构与页面布局

### 4.1 转换首页（两栏工作台）

#### 桌面端（≥ 768px）
- 左栏：操作区
  - 模板选择（卡片/分段控件）
  - 上传区（拖拽 + 文件列表）
  - 主按钮「开始转换」（含 loading）
  - 结果摘要卡（成功/失败/警告）
- 右栏：状态与日志
  - 进度（阶段文案 + 进度条）
  - 统计（条数/店铺数/合计数量/耗时）
  - 日志面板（可复制/清空；可选折叠）
  - 下载/打开输出文件入口（成功后）

#### 移动端（< 768px）
- 变为单列：操作区在上，状态/日志折叠为可展开面板或置于下方。

### 4.2 拆零管理页（表格工作台）
- 顶部：返回 / 标题 / 总数徽章
- 工具条：搜索、拆零筛选、每页条数、（新增/保存）
- 表格：编码（可编辑输入框）、拆零规则（toggle）、创建时间、操作（删除）
- 底部：分页
- 未保存更改提示：
  - 行级：有改动的行高亮/角标
  - 全局：工具条或底部状态条显示「待保存 N 条」，保存按钮同步体现

## 5. 组件拆分与目录结构建议

> 目标：让页面变薄、交互复用、样式一致，同时避免引入大型依赖。

建议新增（或调整）目录：

- `src/ui/`：基础 UI 组件（无业务）
  - `Button` / `IconButton`
  - `Card`
  - `Badge`
  - `Modal`（通用弹窗：版本更新、缺失编码、删除确认）
  - `Toast`（统一提示）
  - `Progress`
  - `EmptyState`
  - `SegmentedControl`（模板选择可用）
- `src/features/convert/`：转换页面业务组件
  - `TemplatePicker`
  - `FileDropzone`
  - `FileList`
  - `StatusPanel`（右栏）
  - `LogPanel`
  - `ResultSummary`
- `src/features/split/`：拆零管理业务组件
  - `SplitToolbar`
  - `SplitTable`
  - `Pagination`（也可上移到 `ui/`）
- `src/lib/`：通用逻辑
  - `apiClient`（axios/fetch 封装、统一错误解析与超时策略）
  - `errors`（将后端错误结构归一为前端可显示文案）
  - `hooks`（如 `useToast`、`useAsync`）

## 6. 视觉规范（Design Tokens）

目前 `src/index.css` 已定义了一组变量。建议做一次“分层重命名 + 补齐状态”的升级，避免 token 含义混杂：

- 颜色（示例命名）
  - `--color-bg`（页面背景）
  - `--color-surface`（卡片/面板）
  - `--color-border`
  - `--color-text` / `--color-text-muted`
  - `--color-primary` / `--color-primary-hover` / `--color-primary-press`
  - `--color-success` / `--color-warning` / `--color-danger`
- 间距：`--space-1..--space-6`
- 圆角：`--radius-sm/md/lg`
- 阴影：`--shadow-sm/md`
- 动效：统一 `--transition-fast/normal`

无障碍基础：
- 统一 `:focus-visible` 样式（清晰可见）
- 按钮/输入的禁用态对比度与鼠标指针一致

（可选后续）暗色模式：保留扩展点（`[data-theme="dark"]`），但不作为本阶段硬性要求。

## 7. 关键交互与状态规范

### 7.1 转换流程
- 主按钮状态：`idle → uploading → converting → done / error`
- 错误分级与呈现
  - 网络/超时：Toast + 右栏错误提示（可重试）
  - 业务错误（缺失编码）：保持现有弹窗能力，但统一 modal 样式与按钮文案
  - 文件格式不匹配：在上传区内联提示 + Toast
- 日志策略
  - 默认展示最新若干条
  - 提供「清空」「复制」动作

### 7.2 拆零管理
- 编辑即标记“待保存”
- 保存失败：保留用户输入，不清空 pending；Toast 提示失败原因
- 删除：统一确认弹窗（Modal）

## 8. 迁移与兼容策略

建议分两步迭代，确保可控：
1) 先落地基础设施：tokens 重整 + `ui/` 基础组件 + `apiClient` 统一封装
2) 再改两页布局与组件拆分：
   - Convert：两栏工作台 + 右栏状态面板
   - Split：工具条/表格/分页的结构梳理 + 未保存更改体验

## 9. 验收标准（可测试）
- Convert 页面：桌面端两栏布局成立；移动端可用且不拥挤；转换中不可重复提交；成功/失败提示一致且可定位问题。
- Split 页面：编辑后“待保存 N 条”准确；保存成功/失败反馈一致；删除确认弹窗一致；移动端能完成搜索/编辑/保存。
- 样式：Button/Input/Modal/Toast/Badge/Card 至少在两页复用；页面无明显样式“跳变/不一致”。

## 10. 风险与权衡
- 两栏布局会增加响应式复杂度：通过断点策略与“右栏折叠/下置”兜底。
- 不引入 UI 框架意味着要自己维护基础组件：通过“只做需要的最小集合”控制范围。
